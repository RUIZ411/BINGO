# SUWEET BINGO V1 - SOOP 대결미션 2차 이벤트 로그 테스트 버전

이 버전은 실제 SOOP/SSAPI 서버 연결 전에, 사이트 내부 테스트 버튼도 실제 API 이벤트와 같은 흐름을 타도록 정리한 2차 버전입니다.

## 포함 기능

- 방 4개 구조: 1번 방 / 2번 방 / 3번 방 / 수힛방
- 방 생성 관리자 인증: 아이디 입력 시 `@suweet.com` 자동 추가
- 5자리 방 입장 코드
- 빙고 생성/체크/복구/수정 모드
- 현상금 기능
- 메모장
- 송출용 화면
- 리셋 빙고 메뉴판: 한개 / 한줄 / 전체
- SOOP 대결미션 테스트 패널
- 2차 이벤트 처리 구조
  - 테스트 버튼이 가짜 API 이벤트를 생성
  - 공통 이벤트 처리 함수로 체크 해제 실행
  - 이벤트 ID 중복 처리 방지
  - 기준 불일치 / 연동 OFF / 아이디 불일치도 로그 기록

## 2차 이벤트 테스트 방식

관리자 패널의 `SOOP 대결미션 테스트`에서 다음을 설정합니다.

1. `연동 사용` 체크
2. 스트리머 아이디 입력
   - 여러 명이면 한 줄에 하나씩 입력하거나 쉼표로 구분
3. `연동 설정 저장`
4. 리셋 빙고 메뉴판에서 `한개 / 한줄 / 전체` 값 설정
5. 빙고판에서 테스트용으로 몇 칸 체크
6. 테스트 스트리머 아이디와 펀딩 개수 입력
7. `테스트 실행`

동작 규칙은 다음과 같습니다.

- 테스트 펀딩 개수 === 한개 값 → 체크되어 있는 칸 중 랜덤 1칸 체크 해제
- 테스트 펀딩 개수 === 한줄 값 → 체크되어 있는 칸이 포함된 랜덤 한 줄 체크 해제
- 테스트 펀딩 개수 === 전체 값 → 전체 칸 체크 해제
- 그 외 값 → 빙고판 변화 없음, 로그만 기록

누적 계산이나 배수 계산은 하지 않습니다. 정확히 일치할 때만 실행됩니다.

## 1차와 달라진 점

1차는 테스트 버튼이 바로 체크 해제 로직을 실행하는 구조였습니다.

2차는 테스트 버튼이 먼저 아래와 같은 이벤트 객체를 만들고, 실제 API 연결 때도 재사용할 공통 처리 함수로 넘깁니다.

```js
{
  eventId: "manual_...",
  source: "manual-test",
  eventType: "battleMissionFunding",
  streamerId: "suheet",
  amount: 300,
  createdAt: 1234567890
}
```

이 구조 덕분에 3차에서 실제 SOOP/SSAPI 이벤트를 받으면 같은 형태로 변환해서 처리할 수 있습니다.

## Firebase Rules

현재 방 코드 방식과 2차 테스트 기능을 사용하려면 Realtime Database Rules는 아래처럼 둡니다.

```json
{
  "rules": {
    "bingoRooms": {
      ".read": true,
      ".write": true
    }
  }
}
```
